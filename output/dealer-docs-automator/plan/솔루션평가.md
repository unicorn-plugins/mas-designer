## 1. B/F 투표 (Benefit × Feasibility)

각 축 1~5점 (5=최상).

| # | 후보 | Benefit (비즈니스 임팩트) | Feasibility (실현가능성) | B+F | 비고 |
|:-:|-------|:---:|:---:|:---:|------|
| 1 | **RTA (Real-time Triage Agent)** | 5 | 4 | **9** | 킹핀 직접 해결, 손실 예방 + 규제 대응 |
| 2 | IDV+ (멀티모달 신원확인) | 4 | 4 | 8 | 위변조 탐지 효과 크지만 정확도 튜닝 필요 |

## 2. 우선순위 매트릭스 (4사분면)

```
        F↑ (실현가능성 높음)
        │
  3★ RegGuard  4 AutoDoc   6 AuditSuite
  (No Brainers)               (Utilities)
        │
────────┼──────── 1★ RTA (No Brainers)
        │         2 IDV+ (B/F 균형)
  5 DealerCopilot
  (Bit Bets / 균형)
        │
        └───────────────────→ B (비즈니스 임팩트 높음)
```

- **No Brainers** (고임팩트·고실현성): **RTA · RegGuard**
- **Bit Bets** (고임팩트·난이도): **IDV+** (VLM 정확도 의존성)
- **Utilities** (저임팩트·고실현성): AutoDoc, AuditSuite
- **Unwise**: 없음 (DealerCopilot은 가치는 있으나 킹핀 우선순위 낮음)

→ **No Brainers 2개(RTA · RegGuard)**가 최종 경합.

## 3. AI 실현 가능성 4요소 평가

### 후보 1. RTA (Real-time Triage Agent)

| 요소 | 점수 | 평가 |
|------|:---:|------|
| 데이터 가용성 | 4/5 | 본사 BSS·블랙리스트·경찰청 데이터 존재, **연계 거버넌스 합의 필요** |

| 운영 지속성 | 4/5 | FP(오탐) 사용자 피드백 루프 필요, 설명가능성 유지 |

## 4. 최적안 결정

### 분석

- **RegGuard(R1 근본원인 직접 해결)** vs **RTA(R2 킹핀 직접 해결)**
- 킹핀 선정 시 **R2가 R1·R3의 트로이 목마**로 기능한다고 합의됨 (`킹핀문제.md` 참조)
- RTA 아키텍처에 **규정 RAG(RegGuard) 에이전트를 병렬 노드로 내장**하면 R1도 자연스럽게 해결
- 즉 **RTA를 핵심 솔루션으로 선정하되, RegGuard·IDV+·AutoDoc·AuditSuite를 RTA 아키텍처의 구성 요소로 편입**

### 선정: **RTA — Real-time Triage Agent (확장형)**

**확장형 구성** (RTA 내부에 타 후보 흡수):
```
RTA (LangGraph 오케스트레이션)
├─ 신원확인 에이전트 ← IDV+
├─ 규정 검증 에이전트 ← RegGuard
├─ 사기 탐지 에이전트 (신규)
├─ 서류 생성 에이전트 ← AutoDoc
├─ 영업사원 설명 에이전트 ← DealerCopilot (부차)
└─ 감사 로깅 ← AuditSuite
```
