# 아이디어 발상 가이드

## 목적

문제 가설과 방향성을 기반으로 창의적인 솔루션 아이디어를 생성하고 수렴함.

## 경로 규약 (공통)

- `{OUTPUT_DIR}`: 소비 플러그인(consuming plugin)의 SKILL이 런타임에 치환하는 프로젝트별 산출
  루트 경로 변수. 예: `output/{project}/plan/`. 본 가이드는 **경로 규약만 선언**, 실제 경로
  결정은 소비자의 책임.
- 모든 단계는 **평면 구조**(`{OUTPUT_DIR}/{파일명}`)로 산출하며, 이벤트 스토밍만 다수 파일이라
  `{OUTPUT_DIR}/es/*.puml` 서브폴더 사용.
- 직전 단계 산출물을 본 단계 입력으로 사용. 파일명은 아래 핸드오프 표 기준 고정.

### 핸드오프 표 (6단계 공통)

| 단계 | 산출 파일 | 소비 단계 |
|------|----------|----------|
| 01 problem-hypothesis | `문제가설.md`, `비즈니스가치.md` | 02, 03 |
| 02 direction-setting | `킹핀문제.md`, `문제해결방향성.md` | 03 |
| 03 ideation | `솔루션탐색.md`, `솔루션후보.md` | 04 |
| 04 solution-selection | `솔루션평가.md`, `솔루션우선순위평가.svg`, `핵심솔루션.md` | 05, 06 |
| 05 event-storming | `es/userflow.puml`, `es/{NN}-{유저플로우명}.puml` | 06 |
| 06 user-stories | `userstory.md` | — (최종) |

---

## 입력 (이전 단계 산출물)

| 산출물 | 파일 경로 | 활용 방법 |
|--------|----------|----------|
| 문제 가설 | `{OUTPUT_DIR}/문제가설.md` | 해결할 문제의 범위 설정 |
| 킹핀 문제 | `{OUTPUT_DIR}/킹핀문제.md` | 아이디어 발상의 핵심 초점 |
| 문제해결 방향성 | `{OUTPUT_DIR}/문제해결방향성.md` | Needs Statement를 아이디어 발상 출발점으로 활용 |

> **타 산업 사례 조사는 본 단계에서 직접 수행**. Steal & Synthesize 기법 적용 시 필요한 경쟁사·유사 도메인 성공 사례는 공개 자료·블로그·논문·벤더 사례집으로 3개 이상 확보하여 활용함.

## 출력 (이 단계 산출물)

| 산출물 | 파일 경로 |
|--------|----------|
| 솔루션 탐색 | `{OUTPUT_DIR}/솔루션탐색.md` |
| 솔루션 후보 | `{OUTPUT_DIR}/솔루션후보.md` |

## 방법론

### AI 활용 패턴 카드 (발산 프레임)

아이디어 발상 시 아래 AI 패턴 카드를 프레임으로 적극 활용. 각 아이디어가 어떤 패턴에 속하는지 1개 이상 태깅.

| 패턴 | 설명 | 예시 |
|------|------|------|
| 분류 (Classify) | 입력 데이터를 범주로 나눔 | 스팸 필터·감정 분석·우선순위 분류 |
| 추천 (Recommend) | 개인 맥락에 맞는 선택지 제시 | 콘텐츠·상품·다음 행동 추천 |
| 생성 (Generate) | 텍스트·이미지·코드 생성 | 초안 작성·요약·리포트 자동 생성 |
| 에이전트화 (Agentify) | 복합 작업 자율 수행 | 예약·조사·협상 자동 수행 |
| RAG (Retrieve-Augment) | 지식 기반 검색·인용 | 사내 문서 Q&A·법령 검색 |
| 멀티모달 (Multimodal) | 이미지·음성·문서 입출력 | 영수증 인식·음성 회의록·PDF 요약 |

### 브레인스토밍 원칙

- **양이 질을 낳는다**: 최대한 많은 아이디어 생성
- **판단 유예**: 모든 아이디어 수용
- **자유로운 사고**: 창의적이고 대담한 아이디어 장려
- **사용자 중심**: 각자의 전문성을 내려놓고 사용자 입장에서 아이디어 도출

### 아이디어 발상 기법

#### 1. SCAMPER 기법

기존 아이디어나 솔루션을 7가지 관점에서 변형하여 새로운 아이디어를 창출함.

| 기법 | 핵심 질문 | 적용 가이드 |
|------|---------|-----------|
| **S - Substitute (대체)** | 무엇을 다른 것으로 대체 가능한가? | 재료·자원·프로세스·사람·장소를 바꿈 |
| **C - Combine (결합)** | 무엇을 결합 가능한가? | 기능·아이디어·목적을 합침 |
| **A - Adapt (적용)** | 다른 분야에서 무엇을 가져올 수 있는가? | 유사 문제의 다른 산업 해결법 적용 |
| **M - Modify (수정)** | 무엇을 변경·확대·축소 가능한가? | 크기·모양·기능의 강도 조절 |
| **P - Put to other uses (다른 용도)** | 다른 방식으로 사용 가능한가? | 다른 시장·고객에게 제공 |
| **E - Eliminate (제거)** | 무엇을 제거 가능한가? | 불필요한 기능·단계·규칙 제거 |
| **R - Reverse (역전)** | 순서를 바꾸거나 역으로 생각하면? | 프로세스 순서 변경·역발상 |

**SCAMPER 활용 가이드:**
- Big Idea 도출 시: **Adapt**, **Combine**, **Reverse** 관점 적극 활용
- Little Win Idea 도출 시: **Eliminate**, **Modify** 관점으로 단순화
- Crazy Idea 도출 시: **Magnify**, **Put to other uses** 관점으로 극단화

#### 2. Steal & Synthesize (훔쳐서 재합성)

다른 산업과 분야의 성공 사례를 분석하고, 우리 문제에 맞게 재조합하는 기법임.

##### Steal (훔치기)

**조사 영역:**
- 게임: 사용자 참여, 보상 체계, 레벨 시스템
- 헬스케어: 개인화, 데이터 추적, 습관 형성
- 금융: 신뢰 구축, 보안, 간편 결제
- 이커머스: 추천 시스템, 구매 전환, 배송 혁신
- 교육: 학습 경로, 피드백, 성취 관리
- 모빌리티: 공유 경제, 실시간 매칭, 위치 기반 서비스
- 엔터테인먼트: 개인화 콘텐츠, 구독 모델, 몰입 경험

**핵심 메커니즘 추출:**
- 무엇이 문제였나?
- 어떻게 해결했나?
- 왜 효과적이었나?
- 핵심 메커니즘은 무엇인가?

##### Synthesize (재합성)

**적용 프로세스:**
1. 유사점 찾기: 우리 문제와 성공 사례의 공통점
2. 차이점 인식: 우리만의 제약사항과 특수성
3. 변형 설계: 핵심 메커니즘을 우리 맥락에 맞게 변형
4. 검증: 우리 사용자에게 실제로 효과적일까?

### 참여 팀원 (9명)

| 약어 | 에이전트 | 역할 관점 |
|------|---------|----------|
| PO | product-owner | 비즈니스 가치/MVP 범위 |
| SP | service-planner | 사용자 경험/서비스 흐름 |
| DE | domain-expert | 도메인 규칙/업계 관행 |
| AR | architect | 기술 실현 가능성 |
| AI | ai-engineer | AI/ML 활용 기회 |
| BE | backend-developer | 백엔드 구현 관점 |
| FE | frontend-developer | 프론트엔드/UX 관점 |
| QA | qa-engineer | 테스트/예외 시나리오 |
| DO | devops-engineer | 운영/확장성 관점 |

### 기대 아이디어 수량
- 팀원당: Big Idea 3개 + Little Win Idea 2개 + Crazy Idea 1개 = **6개**
- 총 기대 수량: 9명 × 6개 = **54개**

### 각 팀원 도출 기준

#### 아이디어 구분
- **Big Idea**: 인력과 비용은 많이 드나 실현되면 비즈니스 기여도가 높은 아이디어
- **Little Win Idea**: 실현가능성이 높은 아이디어
- **Crazy Idea**: 절대 불가능할 것 같지만 실현되면 비즈니스 기여도가 가장 높은 아이디어

- Big Idea 3개
- Little Win Idea 2개
- Crazy Idea 1개

## 1단계: 솔루션 탐색

## 출력 형식
```markdown
# 솔루션 탐색
| 팀원 | 아이디어 구분 | 아이디어 제목 | 아이디어 설명 |
|------|-------------|-------------|-------------|
| {팀원명} | Big Idea | {제목} | {상세 설명} |
```

## 2단계: 솔루션 수렴

### 유사도 평가 기준
- **기능 유사도** (가중치 70%): 해결하려는 문제와 제공 가치의 유사성
- **사용자 경험 유사도** (가중치 30%): 서비스 이용 방식과 인터페이스의 유사성
- **합치기 기준**: 유사도 0.7 이상일 때만 합침

### 유사도 평가 형식
| 아이디어 A | 아이디어 B | 기능 유사도 | 경험 유사도 | 종합 유사도 | 합치기 여부 |
|-----------|-----------|------------|------------|-------------|-----------|

## 품질 기준

### 완료 체크리스트
- [ ] 팀원당 6개 아이디어(Big 3/Little Win 2/Crazy 1) 도출
- [ ] SCAMPER 7가지 관점 체계적 적용
- [ ] Steal & Synthesize로 타 산업 사례 최소 3개 분석
- [ ] 아이디어 설명이 구체적이고 상세함
- [ ] 유사도 평가표 작성 (기능 70%, 경험 30%)
- [ ] 유사도 0.7 이상인 아이디어만 합침
- [ ] Steal & Synthesize에 활용할 타 산업 사례 최소 3개를 본 단계에서 직접 조사·확보

### 정량 기준
- 총 아이디어: 54개 (9명 × 6개)
- 타 산업 사례: 3개 이상
- 유사도 합치기 기준: 0.7 이상

## 주의사항

- 초기 생성 시 자기 검열 금지
- 각자의 전문성을 내려놓고 사용자 관점에서 생각
- SCAMPER 7가지 관점을 체계적으로 적용
- Steal & Synthesize로 타 산업 사례 최소 3개 분석
- 아이디어 설명은 최대한 자세하게 작성
- 유사도 0.7 미만 아이디어를 강제 합치지 않음
