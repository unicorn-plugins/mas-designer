# 문제 가설 정의 가이드

## 목적

비즈니스 도메인·주제에 대해 **핵심 문제 가설과 근본원인**을 도출하고,
문제 해결 시 고객·회사가 얻게 되는 **비즈니스 가치**를 정의함.
AI 활용 시나리오를 전제로 하므로, AI로 해결 가능성이 높은 문제 유형
(정보 비대칭·처리 지연·판단 복잡성·반복 작업·개인화 부재·대량 데이터 요약 등)을
우선 고려함.

## 경로 규약 (공통)

- `{OUTPUT_DIR}`: 소비 플러그인(consuming plugin)의 SKILL이 런타임에 치환하는 프로젝트별 산출
  루트 경로 변수. 예: `output/{project}/plan/`. 본 가이드는 **경로 규약만 선언**, 실제 경로
  결정은 소비자의 책임.
- 모든 단계는 **평면 구조**(`{OUTPUT_DIR}/{파일명}`)로 산출하며, 이벤트 스토밍만 다수 파일이라
  `{OUTPUT_DIR}/es/*.puml` 서브폴더 사용.
- 직전 단계 산출물을 본 단계 입력으로 사용. 파일명은 아래 핸드오프 표 기준 고정.

### 핸드오프 표 (6단계 공통)

| 단계 | 산출 파일 | 소비 단계 |
|------|----------|----------|
| 01 problem-hypothesis | `문제가설.md`, `비즈니스가치.md` | 02, 03 |
| 02 direction-setting | `킹핀문제.md`, `문제해결방향성.md` | 03 |
| 03 ideation | `솔루션탐색.md`, `솔루션후보.md` | 04 |
| 04 solution-selection | `솔루션평가.md`, `솔루션우선순위평가.svg`, `핵심솔루션.md` | 05, 06 |
| 05 event-storming | `es/userflow.puml`, `es/{NN}-{유저플로우명}.puml` | 06 |
| 06 user-stories | `userstory.md` | — (최종) |

---

## 출력 (이 단계 산출물)

| 산출물 | 파일 경로 |
|--------|----------|
| 문제 가설 | `{OUTPUT_DIR}/문제가설.md` |
| 비즈니스 가치 | `{OUTPUT_DIR}/비즈니스가치.md` |

> 입력 산출물은 전제하지 않음. 도메인·주제·대상 고객은 본 단계 시작 시 사용자 입력 또는 최초
> 인터뷰로 확보하며, 필요한 관찰·체험·인터뷰 데이터는 본 단계에서 직접 수집하여 문제 도출의
> 근거로 사용함.

## 사전 조사 지침 (필요 시)

체험·관찰·인터뷰 데이터가 부족한 경우 다음을 본 단계에서 직접 수행하여 문제 도출 근거를 확보함.
- **대상 고객 세그먼트 식별**: 1~3개 주요 세그먼트와 그 맥락 간단 정의
- **Pain Points 수집**: 세그먼트별 반복 불편 3~5개 수집 (공개 리뷰·커뮤니티·사내 채널·1:1
  문의 등 2차 자료로 충분)
- **AI 해결 가능성 태깅**: 각 Pain Point에 "AI로 어떻게 해결 가능한지"(분류·추천·생성·에이전트화·
  RAG·멀티모달 중) 1줄 가설 태깅

## 방법론

### 1. 문제 가설과 근본원인 도출

가장 큰 현상문제 3개를 도출하고 5WHY로 근본원인 도출함.

#### 5WHY 작성 형식

| 단계 | 질문 및 응답 |
|:----:|-------------|
| **WHY1** | 왜 {현상}인가? → {응답} |
| **WHY2** | 왜 {WHY1 응답}인가? → {응답} |
| **WHY3** | 왜 {WHY2 응답}인가? → {응답} |
| **WHY4** | 왜 {WHY3 응답}인가? → {응답} |
| **WHY5** | 왜 {WHY4 응답}인가? → {응답} |

- WHY1은 현상문제에 대한 질문 + 질문에 대한 응답으로 작성
- WHY2~WHY5는 이전 WHY의 응답에 대한 질문 + 질문에 대한 응답으로 작성

#### 다층적 근본원인 검토

근본원인은 아래 4가지 측면으로 검토하여 가장 중요한 측면 한개로 근본원인을 식별함:

| 측면 | 검토 항목 |
|------|----------|
| **사용자 심리적** | 인지 부하, 감정, 동기, 습관, 저항감 |
| **제도/시스템적** | 정책/규정, 프로세스, 권한 구조, 인센티브, 예산/자원, 기존 시스템 제약 |
| **맥락적** | 조직 문화, 업계 관행, 시장 상황, 타이밍 |
| **커뮤니케이션** | 정보 전달, 교육, 피드백 체계 |

#### 핵심 근본원인 식별 형식

| 측면 | 근본 원인 | 중요도 |
|------|----------|:------:|
| 사용자 심리적 | {근본원인} | ★★★★☆ |
| 제도/시스템적 | {근본원인} | ★★★☆☆ |
| 맥락적 | {근본원인} | ★★★★★ |
| 커뮤니케이션 | {근본원인} | ★★★☆☆ |

### 2. 비즈니스 가치 도출

근본원인을 제거하여 문제를 해결했을 때 고객과 회사가 얻게 되는 비즈니스 가치를 각각 3개 이하로 도출함.

```markdown
## 고객측면의 비즈니스 가치
- {비즈니스 가치 1}
- {비즈니스 가치 2}
- {비즈니스 가치 3}

## 회사측면의 비즈니스 가치
- {비즈니스 가치 1}
- {비즈니스 가치 2}
- {비즈니스 가치 3}
```

AI 활용 시나리오이므로 정량 가치(시간 절감·처리량 증가·정확도 향상)와 정성 가치(만족도·접근성·일관성)를 구분하여 기입함.

## 품질 기준

### 완료 체크리스트
- [ ] 핵심 문제 3개 도출 (AI로 해결 가능성 있는 유형 포함 여부 검토)
- [ ] 각 문제별 5WHY 완성 (최소 3단계)
- [ ] 다층적 근본원인 4관점(심리적/제도/맥락적/커뮤니케이션) 검토
- [ ] 비즈니스 가치 도출 (고객/회사 각 3개 이하, 정량/정성 구분)
- [ ] 문제는 고객 관점에서 작성, 솔루션 미포함

### 정량 기준
- 핵심 문제: 3개
- 5WHY: 문제당 최소 3단계
- 근본원인 검토: 4가지 측면
- 비즈니스 가치: 고객/회사 각 3개 이하

## 주의사항

- 문제는 고객 관점에서 작성
- 솔루션은 포함하지 않음
- 5WHY는 최소 3단계 이상
- AI 활용 시나리오이므로, "사람만 할 수 있는 문제"보다 "AI 보조·대체·증강이 유효한 문제"에 집중
