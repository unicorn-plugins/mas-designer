# 03 멀티모달 AI 기반 기능

## 1. 문서 요약

### 1.1 WHY — 왜 문서요약이 필요한가?
정보 과부하 시대, 효율적인 문서 처리를 위한 핵심 기술.

| 용도 | 설명 |
|------|------|
| 뉴스 요약 | 긴 기사를 헤드라인/짧은 요약으로 변환 |
| 문서 분석 | 보고서, 논문, 계약서 핵심 내용 추출 |
| 회의록 요약 | 회의 내용을 액션 아이템으로 정리 |
| RAG 파이프라인 | 검색 문서 요약으로 LLM 컨텍스트 효율화 |

### 1.2 HOW — 어떻게 요약하는가?

#### LLM 기반 요약 (생성적 요약)

| 구분 | 모델 | 특징 |
|------|------|------|
| Cloud LLM | Groq (llama-3.3-70b) | LPU 초고속 추론(456 tok/s), GPU 불필요, 무료 티어 제공 |
| Local LLM | EXAONE-4.0-1.2B (LG) | 한국어/영어 이중언어, 과기정통부 평가 1위 |

#### 특화 요약 모델 (경량/고속)

| 언어 | 모델 | 특징 |
|------|------|------|
| 한국어 | KoBART (SKT AI) | 한국어 요약 특화, 경량(500MB), 빠른 처리 |
| 한국어 | pko-T5 | 논문/도서/레포트 요약 특화 |
| 다국어 | mT5 / mBART | 50개+ 언어 지원, 요약/번역 강점 |
| 영어 | PEGASUS / DistilBART | 뉴스 요약 최적화, 경량화·빠른 추론 |

## 2. PDF 처리

### 2.1 WHY — 왜 PDF 처리가 필요한가?
텍스트, 테이블, 이미지를 추출 → 구조화된 형식으로 변환.

| 용도 | 설명 |
|------|------|
| RAG 데이터 전처리 | PDF를 청킹하여 벡터 DB 저장을 위한 텍스트 추출 |
| 문서 자동화 | 계약서, 보고서, 명세서에서 정형 데이터 추출 |
| 지식 베이스 구축 | 기술 문서, 논문을 마크다운으로 변환하여 검색 시스템 구축 |
| 데이터 마이그레이션 | 레거시 PDF 문서를 최신 포맷으로 변환 |

### 2.2 HOW — 어떻게 처리하는가?

#### AI 기반 PDF 처리

| 도구 | 특징 |
|------|------|
| **Docling (IBM)** | OCR 내장(EasyOCR, 한국어/영어 지원), TableFormer AI로 정밀한 테이블 구조 분석, 복잡한 다단 레이아웃 완벽 변환 |

#### 규칙 기반 PDF 처리

| 도구 | 강점 | 특징 |
|------|------|------|
| **PyMuPDF** | 속도 중심 | 수배~수십 배 빠른 처리 속도, 이미지 추출 최상, pymupdf4llm 확장 제공 |
| **pdfplumber** | 위치 중심 | 좌표 기반 정밀 추출, 경계선 없는 테이블도 추출, 시각적 디버깅 기능 |

> 권장: Docling → RAG/AI 전처리, PyMuPDF → 대용량 처리, pdfplumber → 정밀·구조 중심 추출.

## 3. STT(Speech To Text) + Diarization (화자 인식)

### 3.1 WHY — 왜 STT가 필요한가?
음성 신호를 텍스트 데이터로 변환하여 업무에 활용.

| 용도 | 설명 |
|------|------|
| 스마트 회의록 | 회의 내용 실시간 기록 및 발화자별 정리 |
| 콘텐츠 자막 | 영상 음성 분석 후 자동 자막 생성 및 번역 |
| 콜센터 분석 | 상담원-고객 대화 분리, 서비스 품질 분석 및 감정 인식 |

### 3.2 HOW — 어떻게 처리하는가?

#### LLM 기반 STT + 화자 인식
API 호출만으로 높은 수준의 결과물 획득.

| 모델 | 용도 | 특징 |
|------|------|------|
| **Whisper API** | 간단한 받아쓰기 | 다국어 인식, 영어 번역, 화자 분리 미지원 |
| **LLM Diarization** | 대화 분석/요약 병행 | 멀티모달 LLM 화자 분리, 요약/감정 분석 병행 |

#### 음성 AI 모델 기반 STT + 화자 인식
로컬 GPU 환경, 대용량/보안 중시 프로젝트에 적합.

| 모델 | 용도 | 특징 |
|------|------|------|
| **Whisper + Pyannote** | 일반적 로컬 구축 | 업계 표준 조합, 높은 정확도, 로컬 실행 |
| **WhisperX** | 고성능/대량 처리 | 4~10배 빠른 처리, 단어 수준 타임스탬프 |

> 권장: Whisper API → 받아쓰기, LLM → 대화 분석, Pyannote → 로컬, WhisperX → 대량.

## 4. TTS(Text To Speech)

### 4.1 WHY — 왜 TTS가 필요한가?
텍스트를 억양과 감정까지 표현하는 자연스러운 음성으로 변환.

| 용도 | 설명 |
|------|------|
| 오디오북 제작 | 텍스트 콘텐츠를 음성으로 변환하여 청취형 콘텐츠 생성 |
| 음성 안내 시스템 | 네비게이션, 키오스크, ARS 등 음성 안내 자동화 |
| 다국어 콘텐츠 | 텍스트 번역 후 해당 언어 음성으로 변환 |
| 가상 어시스턴트 | AI 챗봇, 스마트 스피커의 음성 응답 생성 |

### 4.2 HOW — 어떻게 처리하는가?

#### Cloud LLM API 기반
API 호출만으로 고품질 음성 생성. GPU 불필요, 즉시 사용 가능.

| 모델 | 특징 |
|------|------|
| **OpenAI TTS API** | 6가지 음성 스타일(alloy, echo, fable, onyx, nova, shimmer), 다국어 자동 인식, 실시간 스트리밍 지원, tts-1(표준) / tts-1-hd(고품질) |

#### 로컬 모델 기반
GPU 환경에서 직접 모델 실행. API 비용 없음, 데이터 보안 유지.

| 모델 | 특징 |
|------|------|
| **Bark (Suno AI)** | 13개 언어 지원(한국어 포함), 감정/비언어적 표현 생성 가능, MIT 라이선스, 상업적 사용 가능 |
| **VITS (Kakao)** | 109명 다중 화자, End-to-End 고품질, 영국/스코틀랜드/미국 등 다양한 억양, espeak-ng 필수, 영어만 지원 |

#### 용도별 추천 모델

| 용도 | 추천 |
|------|------|
| 한국어 | Kakao VITS, Bark |
| 빠른 속도 | FastPitch* |
| 음성 복제 | XTTS-v2** |
| 감정 표현 | Bark |
| 다국어 | MMS-TTS, Bark |
| 프로덕션 | SpeechT5 |

> \* Non-autoregressive: 모든 출력을 병렬로 동시 생성하여 빠른 속도.  
> \*\* Zero-shot cloning: 짧은 음성 샘플로 새 화자 복제.

## 5. VLM(Vision Language Model)

### 5.1 WHY — 왜 VLM이 필요한가?
이미지를 인식하여 자연어로 설명·질의응답 등 다양한 시각-언어 태스크 수행.

| 용도 | 설명 |
|------|------|
| 문서/OCR 분석 | 문서, 차트, 표의 텍스트 추출 및 구조 이해 |
| 이미지 캡셔닝 & VQA | 이미지 내용 설명, 질문 답변 생성 |
| 품질 검사 | 제조업 제품 결함 탐지 및 불량 판별 자동화 |
| 의료 영상 분석 | X-ray, CT 등 의료 이미지 판독 보조 |

### 5.2 HOW — 어떻게 처리하는가?

#### Cloud LLM API 기반: Google Gemini Vision
API 호출만으로 이미지 분석. GPU 불필요, 즉시 사용 가능.

| 모델 | 특징 |
|------|------|
| **Gemini Vision** | 빠른 프로토타이핑, 다국어 지원(한국어 포함), 1M+ 토큰 컨텍스트, 이미지/비디오/오디오 멀티모달 입력 처리, GPU 불필요 (클라우드 처리) |

#### 로컬 모델 기반: Qwen2.5-VL (Alibaba)
GPU 환경에서 직접 실행. API 비용 없음, 데이터 보안 유지.

| 모델 | 특징 |
|------|------|
| **Qwen2.5-VL-7B** | 보안/대량 처리, 3B~72B 다양한 모델 크기, 8-bit 양자화 지원, OCR 최강 — 문서/표/차트 텍스트 추출 우수, 한국어 지원, 동적 해상도 자동 조절 |

## 6. Function Calling

### 6.1 WHY — 왜 Function Calling이 필요한가?
LLM을 '생각하는 존재'에서 '행동하는 Agent'로 전환하는 기반 기술.

| 한계 | 해결 |
|------|------|
| **정보 단절**: 학습 데이터 이후 정보 접근 불가 | 내·외부 정보 검색으로 정보의 최신성과 정확성 향상 |
| **외부 시스템 연동**: LLM은 생각만 가능, 실제 행동 불가 | 외부 API/시스템 제어 함수 호출하여 행동하는 에이전트로 전환 |
| **단일 턴 한계**: 복잡한 작업의 단계적 수행 어려움 | 반복 호출로 다단계 작업 수행 |

### 6.2 주요 사용 사례

| 분류 | 예시 |
|------|------|
| 정보 조회 | 날씨, DB, 검색 |
| 기능 확장 | 계산기, 코드 실행 |
| 외부 시스템 실행 | 일정, 이메일, 결제 |
| 데이터 추출 | 엔티티, 분류 |

### 6.3 어떻게 동작하는가? — DREAM 워크플로우 (5단계)
**D**efine → **R**equest → **E**valuate → **A**ction → **M**odel response

| # | 단계 | 설명 |
|---|------|------|
| 1 | 도구 정의 | 함수명·설명·파라미터 JSON 스키마 정의 |
| 2 | 요청 전송 | 사용자 메시지 + 도구 목록 API 전송 |
| 3 | 모델 판단 | 모델이 프롬프트 분석 후 함수 호출 결정 |
| 4 | 함수 실행 | 앱에서 실제 함수 호출 수행 |
| 5 | 결과 반환 | 모델에 함수 결과 전달하여 최종 응답 생성 |

### 6.4 벤더별 비교

| 관점 | OpenAI | Claude | Gemini |
|------|--------|--------|--------|
| 용어 | Tool Calling | Tool Use | Function Calling |
| 도구 정의 | JSON Schema를 function 래퍼로 감쌈 → 구조 깊지만 자동완성 용이 | JSON Schema 직접 사용 → 가장 간결 | SDK 타입 클래스 사용 → 타입 안전하지만 코드량 많음 |
| 함수 호출 감지 | `message.tool_calls` | `stop_reason == "tool_use"` | `response.function_calls` |
| 함수 정보 추출 | `arguments`가 JSON 문자열 → `json.loads()` 필요 | `input`이 이미 dict(딕셔너리) → 바로 사용 가능 | `args`가 MapComposite → `dict()` 변환 필요 |
| 결과 반환 | 호출 ID로 매칭 (`tool_call_id`) | 호출 ID로 매칭 (`tool_use_id`) | 함수 이름으로 매칭 (고유 ID 없음) → 동일 함수 병렬 호출 시 결과 반환 순서 주의 |

### 6.5 함수 호출 통제 — tool_choice

LLM의 도구 호출을 개발자가 직접 통제하고 고급 패턴으로 확장해야 실전 서비스가 가능.

활용 시나리오:
- 특정 함수만 호출하도록 제한이 필요한 경우  
- 독립적 요청을 병렬 처리해 응답 속도를 높이는 경우  
- 함수 결과를 이어 붙여 복합 작업을 처리하는 경우  
- 실시간 스트리밍으로 UX와 비용을 개선하는 경우  

#### tool_choice — 함수 호출 제어 옵션

| 모드 | OpenAI | Claude | Gemini |
|------|--------|--------|--------|
| 자동 결정 | `"auto"` | `"auto"` | `AUTO` |
| 강제 사용 | `"required"` | `"any"` | `ANY` |
| 특정 함수 강제 | `{type:"function"...}` | `{type:"tool"...}` | `allowed_fn_names` |
| 사용 금지 | — | `"none"` | `NONE` |

### 6.6 고급 호출 패턴

| 패턴 | 설명 | 예시 | 지원 |
|------|------|------|------|
| 병렬 호출 | 독립 함수를 동시 호출 | 서울+도쿄 날씨 동시 조회 | 3사 모두 지원 |
| 순차 호출 | 앞 함수 결과 → 뒷 함수 입력 | 위치확인 → 날씨조회 | 3사 모두 지원 |
| 스트리밍 | 체감 지연 감소, 대량 응답 처리 효율적 | — | Gemini: LLM 응답만 가능. 함수 호출 결정은 스트리밍 못함 |
